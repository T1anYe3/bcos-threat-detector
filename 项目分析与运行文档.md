# 联盟链多维安全监控体系项目分析与运行文档

## 1. 项目目标与建设范围

本项目目标是构建一个覆盖以下六个维度的安全监控体系：

- 智能合约安全
- 交易模型安全
- 区块处理安全
- 共识安全
- 账户安全
- 网络安全

并实现对联盟链（FISCO-BCOS）的实时安全态势感知、威胁分析与预警，支撑链系统安全、可持续和高效运作。

---

## 2. 项目结构分析

当前工程采用 `Java + Python + SQLite + Web前端` 的分层架构，核心目录与功能如下：

- `src/main/java/org/example/RealTimeMonitor.java`
  - 链上实时采集主程序
  - 获取区块、交易、回执并写入数据库
  - 启动时同步共识白名单
  - 采集语义结果与网络观测指标

- `src/main/java/org/example/DBUtil.java`
  - 数据库初始化与写入工具
  - 核心表：
    - `transactions`：交易主数据
    - `semantic_calls`：语义解码数据
    - `network_metrics`：网络观测指标

- `src/main/java/org/example/NodeInfoCollector.java`
  - 从链上动态拉取共识节点（Sealer）
  - 写入 `nodes` 表，支撑白名单动态适配（换链无需手改配置）

- `src/main/java/org/example/InstructionAnalyzer.java`
  - 语义解码层（调用类型、方法ID、函数名、敏感标记、参数摘要）
  - 支持结构化输出，供检测引擎使用

- `src/main/java/org/example/datadetecter.py`
  - 多阶段检测引擎（规则 + 多模型 + 行为 + 融合）
  - 读取 DB/CSV 两种输入
  - 关联语义与网络特征

- `src/main/java/org/example/app_json.py`
  - 统一后端 API
  - 实时模式：`/api/realtime`
  - 历史CSV导入：`/api/import_csv`
  - 白名单刷新：`/api/refresh_whitelist`

- `src/main/java/org/example/dashboard_json.html`
  - 态势大屏前端
  - 支持实时检测和 CSV 历史检测展示

## 3. 项目与目标匹配程度评估

### 3.1 整体匹配结论

当前项目已经具备“多维安全监控体系”的主体能力，整体可评估为：

- **总体匹配度：中高（约 80%~85%）**
- **可运行性：高（采集、检测、展示链路已贯通）**
- **工程完整性：中（具备实时与历史双模式）**

### 3.2 各安全维度覆盖情况（方法、特征、威胁感知）

- **智能合约安全（中低）**
  - 已有威胁感知措施：
    - `InstructionAnalyzer` 对 `input_data` 做语义解码，区分部署/转账/函数调用
    - 对已识别函数输出 `method_id`、`method_name`、`is_sensitive`、`decode_success`、`arg_summary`
    - 检测引擎中已加入语义规则层（敏感调用失败、未知函数大载荷、解码失败）与语义异常层（方法调用突发）
  - 当前使用方法：
    - 规则检测：阈值与组合规则
    - 异常检测：基于方法调用计数的稳健统计（Robust Z/MAD）
  - 主要体现特征：
    - `method_id`、`method_name`、`is_sensitive`、`decode_success`、`feat_input_len`、`feat_status_code`

- **交易模型安全（高）**
  - 已有威胁感知措施：
    - 多阶段异常检测（规则 + 机器学习 + 融合）
  - 当前使用方法：
    - 规则：Gas 超限、输入长度异常、状态码异常、Gas 比例异常
    - 统计/ML：IsolationForest、LOF、EllipticEnvelope、Autoencoder(MLP 重构)
    - 融合：多检测器加权评分 + 投票门限
  - 主要体现特征：
    - `feat_gas_used`、`feat_input_len`、`feat_nonce`、`feat_gas_ratio`、`feat_status_code`

- **区块处理安全（中高）**
  - 已有威胁感知措施：
    - 区块级突发交易检测
    - 区块失败率风暴检测
  - 当前使用方法：
    - 区块聚合 + 稳健统计（MAD/Robust Z）
  - 主要体现特征：
    - `blocknumber`、`feat_block_trade_count`、`fail_rate`、`feat_status_code`

- **共识安全（中）**
  - 已有威胁感知措施：
    - `NodeInfoCollector` 动态从链上同步 `Sealer` 到 `nodes` 表
    - 检测端以 `nodes` 白名单比对出块节点，识别非法节点来源
  - 当前使用方法：
    - 白名单身份校验（规则）
  - 主要体现特征：
    - `sealer`、`nodes(role='Sealer')`

- **账户安全（中高）**
  - 已有威胁感知措施：
    - 基于发送账户分组的 nonce 序列跳变/回退检测
  - 当前使用方法：
    - 时序行为检测（IQR 阈值 + 异常比例 + TopK 极端样本）
  - 主要体现特征：
    - `from_addr`、`feat_nonce`、`feat_ts`

- **网络安全（中）**
  - 已有威胁感知措施：
    - 在实时采集中记录 `rpc_latency_ms`、`block_interval_ms`、`fail_rate`
    - 在检测端执行网络观测规则，识别节点压力异常
  - 当前使用方法：
    - 基于中位数倍率阈值的规则检测（延迟、区块间隔、失败率）
  - 主要体现特征：
    - `feat_rpc_latency`、`feat_block_interval`、`feat_network_fail_rate`

## 4. 仍需改进的关键点（分维度）

### 4.1 智能合约安全：重点补强项

- **现有威胁感知能力**
  - 已能识别：敏感调用失败、未知方法大载荷、方法调用突发
  - 已能形成：语义规则告警 + 语义异常告警

- **需要补充**
  - ABI 驱动解码：由“手工方法ID匹配”升级为“按合约 ABI 自动解码”
  - 权限语义：引入账户角色（管理员/普通调用者）与函数权限矩阵
  - 函数级基线：按 `contract + method` 维护调用频次/失败率基线
  - 事件级语义：关联 `event logs` 与交易调用形成更完整证据链

- **建议新增检测方法**
  - 规则：`敏感函数 + 非授权调用者`、`敏感函数短时突发`
  - 模型：`contract-method` 维度的时间序列异常（EWMA / Prophet / STL）

### 4.2 网络安全：重点补强项

- **现有威胁感知能力**
  - 已能识别：RPC 时延激增、区块间隔异常、失败率异常
  - 已能形成：网络压力类异常告警

- **需要补充**
  - 节点主机观测：`net_in/out`、`tcp_established`、`syn_rate`、CPU/内存
  - P2P 观测：`peer_count`、`peer_churn`（连接抖动）、断连率
  - 关联判定：网络指标异常与链上失败/拥塞同窗共振提升风险级别

- **建议新增检测方法**
  - 规则：`syn_rate` + `peer_churn` 联合阈值
  - 模型：多变量异常检测（IsolationForest on network feature vector）

### 4.3 交易模型安全：优化建议

- **现有方法**
  - 规则 + IForest + LOF + Elliptic + Autoencoder + 融合

- **补强方向**
  - 按合约类型/业务类型分层建模，降低跨业务分布偏差
  - 引入在线阈值自适应，减少固定阈值带来的误报
  - 将模型输出与语义、网络证据联合解释，提升可解释性

### 4.4 区块处理安全：优化建议

- **现有方法**
  - 区块突发与失败风暴（MAD/Robust Z）

- **补强方向**
  - 引入“连续异常区块窗口”检测，避免单点噪声
  - 增加空块率、出块偏斜、确认延迟等治理性指标

### 4.5 共识安全：优化建议

- **现有方法**
  - 链上动态白名单 + 非法节点识别

- **补强方向**
  - 增加节点出块占比异常检测（权重偏斜）
  - 增加节点可用性趋势（长周期稳定性）与节点信誉评分

### 4.6 账户安全：优化建议

- **现有方法**
  - nonce 跳变/回退行为检测

- **补强方向**
  - 增加账户图谱特征（扇出、扇入、短周期关联地址簇）
  - 增加“新地址爆发 + 高频交互”联合规则，识别脚本化攻击

## 5. 项目运行方法

## 5.1 环境准备

在项目根目录 `F:\idea\IdeaProjects\fisco` 执行：

1. 准备 Java 11+ 与 Maven  
2. 准备 Python 3.10+  
3. 安装 Python 依赖：

```powershell
pip install flask flask-cors pandas numpy scikit-learn
```

> 说明：数据库文件默认使用项目根目录下 `blockchain_data.db`。

## 5.2 启动顺序（推荐）

### 终端A：启动链上实时采集

方式1（IDEA）：
- 直接运行 `RealTimeMonitor.main()`

方式2（命令行）：

```powershell
mvn -DskipTests exec:java "-Dexec.mainClass=org.example.RealTimeMonitor"
```

该进程负责：
- 同步白名单（`nodes`）
- 写入交易（`transactions`）
- 写入语义（`semantic_calls`）
- 写入网络观测（`network_metrics`）

### 终端B：启动检测后端与可视化

```powershell
python src/main/java/org/example/app_json.py
```

启动后访问：

- [http://127.0.0.1:5000](http://127.0.0.1:5000)

## 5.3 运行接口说明

- `GET /api/realtime`：实时检测结果
- `POST /api/import_csv`：导入历史 CSV 执行检测
- `POST /api/refresh_whitelist`：手动刷新链上白名单

## 5.4 常见问题排查

- 页面有但无数据：
  - 检查 `RealTimeMonitor` 是否正在运行
  - 检查链连接配置 `src/main/resources/config.toml`

- 白名单异常：
  - 调用 `POST /api/refresh_whitelist`
  - 检查 `nodes` 表是否有 `Sealer` 数据

- IDEA 编译缓存问题：
  - `Build -> Rebuild Project`

- Python 依赖错误：
  - 重新执行 `pip install` 安装缺失包
